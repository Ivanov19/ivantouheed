So, What Is Machine Learning Again?

Jul 2026

Everyone's talking about it. Most explanations either drown you in math or reduce it to "computers that think." Neither is particularly useful. Let's try something more honest.

The simplest way to put it

Machine learning is a way of getting a computer to find patterns in data without explicitly programming every rule.

In traditional programming, you write the rules. You tell the computer: if temperature is above 100°C and pressure exceeds this threshold, flag it as a problem. You encode the logic. The computer executes it.

In machine learning, you flip that. You give the computer examples — here are 10,000 situations, here's what happened in each one — and let it figure out the rules itself. You don't write the logic. You show it enough cases and it learns to generalize.

That's it. That's the core idea. Everything else is detail.

What "the model learns" actually means

This phrase gets thrown around a lot and it's worth slowing down on.

A model is just a mathematical function — something that takes inputs and produces an output. What "learning" means, technically, is adjusting the internal parameters of that function until its outputs match the data you've shown it as closely as possible. It's optimization, not some cognitive-learning. The model doesn't understand any of your real data you put in. It's finding the numerical configuration that minimizes error on the examples given to it.

This distinction matters though, because it tells you exactly where ML breaks down: give it examples that don't represent what you actually care about, and it will optimize perfectly for the wrong thing. It has no way to know the difference.

Two broad types worth knowing

Most ML falls into either of the two categories.

Supervised learning — you have inputs and you have known outputs. You train the model on pairs of (input → output) and ask it to predict outputs for inputs it hasn't seen before. Predicting house prices, classifying emails as spam, estimating a material's performance from its composition — all supervised.

Unsupervised learning — you only have inputs, no labels. You're asking the model to find structure in the data on its own. Grouping customers by behavior, detecting anomalies, compressing data. No right answer to compare against — you're exploring.

Most of what gets called "practical ML" in industry and research is supervised. That's where we'll spend most of this series.

[There are several other categories as Reinforcement, semi-supervised or self-supervised learning. But we are not going to cover those in our "The Basics of ML Series" here. If you are further interested, check this out!]

Regression vs Classification — one more split worth making

Within supervised learning, there's an important distinction based on what you're predicting.

Regression — the output is a continuous number. How much? How long? What value? Predicting temperature, price, or a performance score are regression problems.

Classification — the output is a category. Which one? Is it this or that? Spam or not spam, cat or dog, disease present or absent — classification.

The same underlying logic applies to both. The math at the end changes slightly. The pipeline — data, preprocessing, training, validation — is mostly identical.

Where does an ML project actually start?

Not with the model. That's the part most people get wrong.

It starts with the data. Specifically, with understanding what you have, what's missing, what's noisy, and whether the examples you're working with are actually representative of the problem you want to solve.

A sophisticated model trained on poorly understood data will give you sophisticated-looking wrong answers. A simple model trained on clean, well-understood data will often outperform it.

The model is the last decision you make, not the first.
